Fase 1 — Crashes e Panics

Bugs que produzem panic no compilador ou segfault em runtime. Alta prioridade — afetam código que a doc promete funcionar.
P-1 Fix nil crash em métodos de String no codegen concluído
Diagnóstico

Em gen_exprs.go, a função generateMethodCall tenta métodos via primitiveMethodCFunc (contains, toUpper, toLower) e chama g.findFunc(cfn) imediatamente. Se a função C não está declarada no módulo LLVM atual (não há extern fn no arquivo), findFunc retorna nil e g.current.NewCall(nil, args...) causa panic. O prelude registra esses métodos via classe StringExtensions, mas o caminho primitiveMethodCFunc roda antes e não testa nil.

Arquivo

internal/codegen/gen_exprs.go

Reprodução
// crasha no compilador:
val s = "hello"
print(s.toUpper())  // panic nil ptr
print(s.contains("x"))  // idem
Fix
// Antes (linha ~1392):
if cfn := primitiveMethodCFunc(bt.Name, me.Property); cfn != "" {
    callArgs := append([]value.Value{obj}, args...)
    return g.current.NewCall(g.findFunc(cfn), callArgs...), nil
}

// Depois: checar nil antes de chamar
if cfn := primitiveMethodCFunc(bt.Name, me.Property); cfn != "" {
    if fn := g.findFunc(cfn); fn != nil {
        callArgs := append([]value.Value{obj}, args...)
        return g.current.NewCall(fn, callArgs...), nil
    }
    // fn não está no módulo; continua para StringExtensions
}
Teste de aceitação
val s = "hello"
val up = s.toUpper()   // "HELLO"
print(s.contains("ell"))  // true
print("abc".toLower())    // "abc"
P-2 break / continue como expressão em arm de match concluído
Diagnóstico

O parser de arms de match (parse_patterns.go / parse_exprs.go) não aceita break nem continue como expressão de arm. O token BREAK não está listado no conjunto inicial de parseExpr quando chamado dentro de um arm. Isso bloqueia o idioma canônico de canal: None => break.

Arquivo

internal/parser/parse_exprs.go

Reprodução
while !ch.isClosed() {
    match ch.recv() {
        Some(v) => print(v)
        None    => break  // parse error
    }
}
Fix
  • Em parseMatchArm (ou onde o corpo do arm é parseado), adicionar case para BREAK e CONTINUE que retornam *BreakStmt / *ContinueStmt.
  • O codegen já lida com BreakStmt em blocos de loop — apenas garantir que o nó chegue até ele via match arm.
Teste de aceitação
while cond {
    match valor {
        Some(v) => processar(v)
        None    => break
        _       => continue
    }
}
P-3 loop como expressão retornando valor via break valor concluído
Diagnóstico

generateLoopStmt em gen_control.go retorna sempre nil, nil — sem suporte a valor de retorno. O codegen de generateVarDecl tenta armazenar esse nil e crasha com nil pointer em gen_exprs.go:485. Também precisa que BreakStmt no AST carregue um valor opcional.

Arquivos

internal/parser/ast.go
internal/parser/parse_stmts.go
internal/codegen/gen_control.go

Plano de implementação
  • AST — adicionar campo Value ast.Expr em BreakStmt (opcional, nil = break simples).
  • Parser — em parseBreakStmt, se o token seguinte não é ; / } / newline, parsear uma expressão como valor do break.
  • Codegen — em generateLoopStmt: alocar um slot (alloca) para o valor do loop antes do corpo; em cada BreakStmt com valor, fazer store no slot e saltar para after; após o loop, load do slot e retornar o valor.
  • Checker — inferir o tipo do loop a partir dos valores de break presentes no corpo.
Teste de aceitação
var t = 0
val resultado = loop {
    t = t + 1
    if t == 5 { break t * 10 }
}
print(resultado)  // 50

val conexao: Result[Conexao] = loop {
    match tentarConexao() {
        Ok(c)  => break Ok(c)
        Err(e) => { tentativas = tentativas + 1 }
    }
}
P-4 Segfault ao chamar função via parâmetro de tipo-função concluído
Diagnóstico

Em generateCallExpr (linha ~957), quando o callee é um parâmetro do tipo (Int) -> Int, ele é representado como i8* (closure fat-pointer) — mas o código tenta tratá-lo como *funcType diretamente via ptrType.ElemType.(*types.FuncType), que falha silenciosamente e cai em callClosureI8Ptr com um objeto mal-formado, causando segfault. O problema está em como funções nomeadas passadas como argumento são empacotadas (ou não) em closure.

Arquivo

internal/codegen/gen_exprs.go

Reprodução
fn aplicar(f: (Int) -> Int, x: Int) -> Int {
    return f(x)  // segfault
}
aplicar(dobrar, 5)
Plano de implementação
  • Investigar como dobrar é empacotado ao ser passado como argumento: deve gerar um closure wrapper { fn_ptr, env=null }.
  • Em generateCallExpr: quando o checker informa que o callee é FuncType e o valor LLVM é um i8*, usar callClosureI8Ptr com o layout correto (extrair fn_ptr do slot 0 do closure).
  • Garantir que ao passar uma função nomeada como valor, ela seja embrulhada em um closure mínimo { ptr_to_func, null_env } compatível com callClosureI8Ptr.
  • Cobrir também o caso de .map(fn_nomeada) e .filter(fn_nomeada).
Teste de aceitação
fn dobrar(x: Int) -> Int = x * 2
fn aplicar(f: (Int) -> Int, x: Int) -> Int = f(x)
print(aplicar(dobrar, 5))   // 10
val ds = [1,2,3].map(dobrar)  // [2,4,6]

Fase 2 — Sintaxe Ausente no Parser

Features documentadas que o parser rejeita. Bloqueiam código válido segundo a spec.
P-5 Args nomeados em construtores de enum e destructuring de record em match concluído
Diagnóstico

O parser trata Ctor(a: v, b: w) como chamada de função e rejeita o COLON em posição de argumento. Também não parseia Record { field1, field2 } como padrão de destructuring em arm de match (confunde com declaração de tipo). Dois problemas relacionados no mesmo parser de expressões/padrões.

Arquivos

internal/parser/parse_exprs.go (chamadas)
internal/parser/parse_patterns.go (match arms)

Parte A — Args nomeados em construtores
  • Em parseCallArgs: detectar o padrão ident COLON expr e parsear como NamedArg{Name, Value}.
  • No AST de CallExpr: adicionar suporte a NamedArgs []NamedArg opcionalmente.
  • No checker: validar nomes contra os campos do enum, reordenar conforme definição.
  • No codegen: empacotar argumentos na ordem dos campos do enum, ignorando os nomes (já corrigidos pelo checker).
enum Forma { Circulo(raio: Float)  Retangulo(w: Float, h: Float) }
val c = Forma.Circulo(raio: 5.0)       // OK
val r = Forma.Retangulo(w: 10.0, h: 4.0) // OK
Parte B — Record destructuring em match
  • Em parsePattern: quando se vê Ident LBRACE, parsear { field1, field2, ... } como RecordPattern.
  • No checker: verificar que os campos existem no record/classe.
  • No codegen: extrair cada campo por índice de struct.
match ponto {
    Ponto { x, y } => print("$(x), $(y)")
}
P-6 Tipo explícito em [] em contexto de expressão concluído
Diagnóstico

O parser só aceita T[A, B] em contexto de tipo (declarações). Em contexto de expressão, [ sempre abre um array literal. Isso bloqueia: instanciação de genérico com tipo explícito (Par[Int, Int] { ... }), chamada de função genérica explícita (identidade[Bool](x)), e construtores de enum genérico multilinha (Arvore.No(\n valor: 1,\n ...)).

Arquivo

internal/parser/parse_exprs.go

Casos afetados
  • Par[Int, Int] { primeiro: 10, segundo: 20 }
  • Repositorio[Usuario] { items: [] }
  • identidade[Bool](true)
  • Arvore.No(\n valor: 1, ... com newline = SEMICOLON
Plano de implementação
  • Lookup-ahead em parsePrimary: quando o token atual é um Ident e o próximo é [, tentar parsear como instanciação genérica antes de tentar array literal. Usar backtracking ou heurística: se após [ aparecer um tipo (ident maiúsculo, tipo primitivo) seguido de , ou ], é parâmetro de tipo; caso contrário, é array.
  • AST: CallExpr e RecordLiteral devem carregar TypeArgs []Type opcionais.
  • Newline em args de enum multilinha: suprimir inserção de semicolon automático quando dentro de parênteses de chamada (já deve ser o caso — investigar se skipSemicolons está sendo chamado corretamente).
  • Checker: usar TypeArgs para especializar o genérico em vez de inferir.
Teste de aceitação
val coords = Par[Int, Int] { primeiro: 10, segundo: 20 }
val repo   = Repositorio[Usuario] { items: [] }
val z      = identidade[Bool](true)
val arvore = Arvore.No(
    1,
    Arvore.No(2, Arvore.Folha, Arvore.Folha),
    Arvore.Folha
)

Fase 3 — Checker e Semântica de Tipos

Features que chegam ao checker mas produzem erros de tipo incorretos ou semântica incompleta.
P-7 extend — self type no checker e dispatch no codegen pendente
Diagnóstico

O checker tem checkExtendDecl e checkExtendMethodBody, mas ao verificar o corpo dos métodos, não instala selfType corretamente — produz "self usado fora de um método de classe". Além disso, os métodos registrados via extend não são encontrados na resolução de chamadas sobre instâncias (u.saudar() → "no member saudar").

Arquivos

internal/checker/check_decls.go (checkExtendMethodBody)
internal/codegen/gen_exprs.go (generateMethodCall)

Parte A — Checker: selfType em extend
  • Em checkExtendMethodBody: definir c.selfType com o tipo resolvido de n.TypeName antes de verificar o corpo.
  • Registrar os métodos do bloco extend na tabela do tipo alvo (seja ClassType.Methods, BasicType ou extendSelfTypes) para que chamadas sobre instâncias os encontrem.
Parte B — Codegen: dispatch de extend
  • Em generateMethodCall: para tipos básicos (String, Int) e classes, consultar g.extensionMethods[typeName][method] — já existe, mas verificar se está sendo populado corretamente a partir de ExtendDecl.
  • Para classes: adicionar busca em extensionMethods antes de lançar erro "no member".
Teste de aceitação
extend String {
    pub fn exclamar(self) -> String = "$(self)!"
    pub fn repetir(self, n: Int) -> String { ... }
}
print("olá".exclamar())      // "olá!"
print("ab".repetir(3))       // "ababab"

class Usuario { val nome: String }
extend Usuario {
    pub fn saudar(self) = print("Olá, $(self.nome)")
}
val u = Usuario { nome: "Vand" }
u.saudar()  // "Olá, Vand"
P-8 Inferência de tipo em classe genérica sem parâmetro explícito pendente
Diagnóstico

Repositorio { items: [] } falha com "unknown type: T" porque o checker não consegue inferir T a partir do campo items: [] (lista vazia sem hint). Records genéricos como Par { primeiro: "a", segundo: 42 } já funcionam porque os valores carregam tipo concreto. O caso-problema é quando o campo inicializador não tem tipo deduzível (lista vazia, None, etc.).

Arquivo

internal/checker/check_generics.go

Plano
  • Quando instanciar uma classe genérica sem TypeArgs explícitos, coletar os tipos dos campos fornecidos e tentar unificar com os parâmetros genéricos da definição.
  • Para campos cujo inicializador é ambíguo ([], None), propagar o tipo esperado do campo da classe como hint ao verificar a expressão.
  • Se após a inferência ainda houver parâmetros não resolvidos, emitir erro pedindo anotação explícita.
Teste de aceitação
class Repositorio[T] {
    var items: List[T] = []
    pub fn adicionar(self, item: T) { self.items.add(item) }
}
// Caso A: inferência via uso posterior
val repo = Repositorio[Int] { items: [] }  // com P-6
repo.adicionar(42)

// Caso B: sem tipo explícito — inferência do primeiro adicionar
val repo2 = Repositorio { items: [] }
repo2.adicionar(42)  // T inferido = Int
P-9 Enum genérico recursivo — substituição de tipo no checker pendente
Diagnóstico

Arvore.No(1, Arvore.Folha, Arvore.Folha) falha: "incompatible argument 2: expected Arvore[Int], got Arvore[T]". O checker especializa o construtor No com T=Int para o primeiro argumento, mas os parâmetros esq e dir ainda carregam o tipo não-substituído Arvore[T] em vez de Arvore[Int].

Arquivo

internal/checker/check_generics.go

Plano
  • Em specializedEnumVariant (ou equivalente): ao especializar o construtor com a substituição T=Int, aplicar a substituição recursivamente em todos os parâmetros do construtor, incluindo os auto-referenciais (Arvore[T] → Arvore[Int]).
  • Garantir que a substituição é idempotente para evitar loop infinito em tipos recursivos.
Teste de aceitação
enum Arvore[T] {
    Folha
    No(valor: T, esq: Arvore[T], dir: Arvore[T])
}
fn profundidade[T](a: Arvore[T]) -> Int {
    return match a {
        Folha       => 0
        No(_, e, d) => 1 + max(profundidade(e), profundidade(d))
    }
}
val t = Arvore.No(1, Arvore.No(2, Arvore.Folha, Arvore.Folha), Arvore.Folha)
print(profundidade(t))  // 2

Fase 5 — Qualidade de Vida

Não são crashes nem features ausentes — mas tornam a DX consideravelmente melhor.
P-11 Impressão de List e Map via interpolação pendente

print("lista: $(lista)") imprime endereço de ponteiro em vez do conteúdo. Implementar soyuz_list_to_string no runtime e conectar à interpolação quando o tipo for List[T] ou Map[K,V].

Teste de aceitação
val xs = [1, 2, 3]
print("$(xs)")   // "[1, 2, 3]"
print(xs)        // "[1, 2, 3]"
P-12 Métodos sem pub acessíveis dentro da própria classe pendente

Métodos privados (fn sem pub) devem ser chamáveis por outros métodos da mesma classe via self.metodo() — hoje são bloqueados mesmo internamente. Ajustar a verificação de visibilidade no checker para permitir acesso via self dentro do escopo da classe.

P-13 Aviso de parâmetro default com overloading pendente

Quando uma função tem parâmetro default e também overloads com when, garantir que o despacho considera o default antes de tentar o catch-all. Documentar o comportamento esperado nos testes.

  8. Consulte os arquivos de teste existentes em internal/checker/ e internal/codegen/ para entender os padrões de teste do projeto. Em caso de dúvida sobre comportamento esperado, pergunte ao usuário.
